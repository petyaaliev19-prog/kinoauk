# Идеи Развития

## Правила этого бэклога

- Задачи выполняются строго сверху вниз: AI берёт только первую задачу `READY`, у которой закрыты зависимости.
- `WAITING_FOR_APPROVAL` означает стоп: сначала показать пользователю перечисленный артефакт и получить явное «да».
- Одна задача — один законченный, проверяемый результат. Нельзя перепрыгивать через этап ради более эффектной части.
- После каждой задачи исполнитель обязан обновить её статус, решение, изменённые файлы, проверки и короткий handoff для следующей сессии.
- Перед любой работой обязательно прочитать `docs/AI_CONTEXT.md` и `docs/ARCHITECTURE.md`.

## Активный бэклог: умное колесо TMDb

### TMDB-00 — Утвердить два режима и переход между ними

**Статус:** `DONE`
**Зависимости:** нет.
**Цель:** не встраивать TMDb в существующую вкладку жанров, а создать ясный переход между двумя самостоятельными пространствами приложения.

**Предложенный контракт:**

```text
«Домашняя полка»  → нынешнее колесо, личный каталог, жанровый аукцион и ставки
«Прокат по запросу» → поиск TMDb, полный внешний пул и колесо / VHS-автомат
```

**Нужно согласовать с пользователем:**

- как выглядит переключатель: вкладка на пульте, кассета в меню или отдельная кнопка «Выйти в прокат»;
- остаётся ли фон комнаты общим, а меняется только рабочая панель, или у «Проката» появляется свой угол комнаты;
- действие при возвращении: сохранить последний внешний запрос или всегда возвращать к чистой форме;
- можно ли добавить фильм из внешнего пула на «Домашнюю полку» отдельным явным действием.

**Критерии приёмки:** одобренный статический макет перехода и записанное правило изоляции состояния. В функциональный код пока не вносится.

**Решение:** переключатель называется «Полка / Прокат». «Прокат» заменяет центральное колесо статичной VHS-панелью будущего каталога, а правая панель — статичным списком будущего полного пула. Домашние вкладки и список полностью скрыты, а не приглушены. Возврат в «Полку» сохраняет весь личный аукцион. В этом прототипе нет сети и нет смешения состояния.

**Изменённые файлы:** `index.html`, `app.js`, `styles.css`, `test/project.test.js`, `docs/AI_CONTEXT.md`, `docs/ROADMAP.md`.

**Проверки:** `npm.cmd run check` — 36/36 тестов прошли.

**Handoff:** режим хранится в `kinoauk.mode.v1`; `setAppMode()` меняет только видимый режим. Следующая AI-сессия не должна подключать TMDb, пока пользователь не утвердит TMDB-01.

### TMDB-01 — Утвердить контракт «полной выдачи»

**Статус:** `DONE`
**Зависимости:** TMDB-00.
**Цель:** зафиксировать, что запросы вроде «боевик + Брэд Питт» используют весь подходящий набор TMDb, а не скрытую сокращённую подборку.

**Нужно согласовать с пользователем:**

- что считается совпадением: фильм обязан одновременно соответствовать всем выбранным фильтрам (`И`);
- включаем ли сериалы отдельным переключателем, по умолчанию — только фильмы;
- правило перехода: малый набор показывается настоящим колесом, большой — выбирается честно из всего пула VHS-автоматом;
- порог перехода предлагается подобрать после прототипа, а не придумывать вслепую.

**Критерии приёмки:** решение записано в разделе «Утверждённые решения» ниже; в коде ничего не меняется.

**Решение:** фильтры в «Прокате» работают по принципу `И`: результат обязан одновременно соответствовать всем выбранным условиям. Полная выдача TMDb не обрезается скрыто. По умолчанию ищем только фильмы; сериалы добавляются отдельным явным чекбоксом «Учитывать сериалы». Если пул слишком большой для честного читаемого колеса, выбор делается равновероятно из всего пула отдельным VHS-автоматом, а полный список остаётся доступен справа. Точный порог перехода выбирается после прототипа.

**Изменённые файлы:** `docs/ROADMAP.md`.

**Проверки:** не запускались — изменён только бэклог/контракт.

**Handoff:** TMDB-01 утверждён. Следующая задача — TMDB-02: спроектировать локальную модель данных и серверные API-границы без подключения UI к реальному TMDb.

### TMDB-07 — Сбор полного пула TMDb

**Статус:** `DONE`
**Зависимости:** TMDB-03, TMDB-04, TMDB-05, TMDB-06.
**Цель:** по выбранным фильтрам получить все страницы TMDb и сохранить полный пул как отдельный сеанс.

**Старт:** 2026-06-27. Допущение: серверный API можно расширить сразу маршрутами `POST /api/rental/sessions` и `GET /api/rental/sessions/:id`; UI пока не подключаем.

**Критерии приёмки:** «Боевик + выбранный Brad Pitt» выбирает только элементы, удовлетворяющие обоим условиям; при `includeTv=false` в пуле только фильмы; при `includeTv=true` фильмы и сериалы смешаны в одном пуле с явным `media_type`; страницы не обрезаются; пользователь видит итоговое число; все участники пула доступны для просмотра; есть тесты пагинации и дедупликации.

**Решение:** в `server.js` добавлены `POST /api/rental/sessions` и `GET /api/rental/sessions/:id`. Сервер валидирует фильтры, требует минимум один основной фильтр (жанр, актёр или режиссёр), затем собирает полный пул двумя путями: по выбранному человеку через его полные `movie_credits` / `tv_credits` с локальной фильтрацией, а без человека — через `discover/{movie|tv}` с проходом по всем страницам. Пул дедуплицируется по `(tmdb_id, media_type)`, сохраняется в `media`, `media_genres`, `rental_sessions`, `rental_session_items` и читается обратно отдельным endpoint. При `includeTv=true` фильмы и сериалы остаются в одном пуле с явным `media_type`.

**Изменённые файлы:** `server.js`, `tmdb-client.js`, `test/tmdb-client.test.js`, `docs/ROADMAP.md`.

**Проверки:** `npm.cmd run check` — 58/58 тестов прошли.

**Handoff:** следующая задача — TMDB-08. Нужно реализовать честный выбор: для малого пула колесо, для большого — VHS-автомат с равной вероятностью для каждого элемента уже сохранённого пула. Не менять правила полного списка и не подмешивать «Полку».

### TMDB-08 — Честный выбор: колесо или VHS-автомат

**Статус:** `DONE`
**Зависимости:** TMDB-06, TMDB-07.
**Цель:** для небольших пулов оставить колесо, для больших — выбрать один фильм равновероятно из полного пула через отдельный VHS-сценарий.

**Старт:** 2026-06-27. Допущение: серверный выбор можно реализовать до UI как `POST /api/rental/sessions/:id/select`; порог малого пула задаём константой и покрываем тестами, UI-анимация останется следующим слоем.

**Критерии приёмки:** в большом пуле вероятность каждого фильма одинакова; интерфейс показывает размер полного пула и номер выбранной кассеты; результат можно проверить через список пула; ставки либо имеют определённое правило, либо явно отключены в этом режиме по согласованному решению.

**Решение:** в `server.js` добавлен `POST /api/rental/sessions/:id/select`. Выбор идёт только из сохранённых `rental_session_items`, индекс считается как `floor(random * totalCount)`, поэтому каждый элемент полного пула адресуем с одинаковой вероятностью. Сервер сохраняет `selected_media_id`, `selected_at`, `selection_mode` и возвращает `selectedItem`, `selectedIndex`, `totalCount`. Режим выбирается через `selectionModeForPool()`: до 24 кассет — `wheel`, больше — `vhs_machine`. Порог вынесен в `DEFAULT_RENTAL_WHEEL_LIMIT`, чтобы после живой проверки UI его можно было быстро поправить.

**Изменённые файлы:** `server.js`, `test/tmdb-client.test.js`, `docs/ROADMAP.md`.

**Проверки:** `npm.cmd run check` — 61/61 тест прошёл.

### TMDB-09 — Будущая «наша видеотека»: просмотрено Максимом и Олей

## Утверждённые решения

- Для поиска по TMDb берём **полную** выдачу, соответствующую фильтрам; скрыто сокращать её нельзя.
- Если выдача слишком велика для честно читаемого колеса, результат выбирается равновероятно из **всего** пула отдельным VHS-автоматом. Полный список остаётся доступен пользователю.
- Видеотека растёт от реальных запросов и импортов; мы не скачиваем весь мировой каталог заранее.
- Текущий личный аукцион и «Прокат по запросу» — разные режимы. У них не смешиваются активные пулы, ставки и история выбора; переход между ними должен быть явным.
- По умолчанию «Прокат» ищет только фильмы; сериалы подключаются отдельным чекбоксом «Учитывать сериалы», чтобы не размывать обычный вечерний выбор.
- Первый «Прокат» ищет только по TMDb. «Наша видеотека» — отдельная будущая функция на базе двух списков Кинопоиска «Просмотрено»: Максим и Оля.
- В «Прокате» актёрский фильтр означает только актёрское участие; режиссёр — отдельный фильтр. Человека нужно выбрать явно из подсказок TMDb до сбора полного пула.

## AI-оракул маскота

Статус: отложено, не входит в текущий MVP.

Цель: маскот ведёт осмысленные диалоги о выборе фильма, а не только отвечает заранее написанными репликами.

Предпочтительный сценарий:

- Мгновенные реакции на вращение, жанры и победу остаются локальными и заранее поставленными.
- Отдельная компактная кнопка «Спросить оракула» открывает короткий диалог только по явному действию Максима или Оли.
- Оракул получает минимальный контекст: текущий жанровый фильтр, фильмы в колесе, последнего победителя и несколько последних сообщений.
- У оракула сохраняется характер домашнего VHS-комментатора, но он не должен менять результат колеса или обходить анти-демократию.

Варианты реализации:

1. Локальная модель через Ollama: приватно, без API-ключа, но требует установки модели и ресурсов компьютера.
2. Внешний LLM API через новый серверный маршрут `/api/mascot`: выше качество и обычно быстрее, но нужен ключ и есть небольшие расходы. Ключ хранится только на локальном сервере, никогда не в браузере.

Перед началом работы выбрать один из вариантов, продумать хранение истории и добавить отдельные тесты для серверного маршрута и деградации без модели/сети.
